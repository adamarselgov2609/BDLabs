# Лабораторная работа №4
## Исследование индексирования и оптимизации запросов в PostgreSQL

**Дисциплина:** Базы данных  
**Учебный год:** 2025-2026, весна  
**Вариант:** 2 (Airbnb — сервис бронирования жилья)

---

## 1. Цель работы

Провести экспериментальное исследование эффективности индексирования в PostgreSQL и обосновать выбор оптимальных индексов для различных типов запросов на основе анализа планов выполнения и характеристик данных.

---

## 2. Схема базы данных

База данных сервиса бронирования жилья состоит из 5 таблиц:

| Таблица | Назначение | Связи |
|---------|------------|-------|
| **users** | Пользователи (гости, хосты, админы) | Родительская для listings, bookings |
| **listings** | Объявления жилья | Дочерняя к users (host_id) |
| **bookings** | Бронирования | Дочерняя к users (guest_id) и listings (listing_id) |
| **reviews** | Отзывы | Дочерняя к bookings |
| **payments** | Платежи | Дочерняя к bookings |

**ER-связи:** users 1→N listings, users 1→N bookings, listings 1→N bookings, bookings 1→0..1 reviews, bookings 1→0..1 payments.

## 4. Исследуемые запросы

---

### Запрос 1: Сложный фильтр

**SQL:**
```sql
EXPLAIN ANALYZE
SELECT * FROM listings
WHERE city = 'Москва'
AND property_type = 'apartment'
AND price_per_night BETWEEN 3000 AND 10000
AND is_active = TRUE;
```

**Гипотеза:**
Существующий индекс `idx_listings_city` помогает только по городу, остальные условия — `Filter`. Составной индекс по всем полям `WHERE` ускорит запрос.

**Новый индекс:**
```sql
CREATE INDEX idx_listings_city_type_price_active 
ON listings (city, property_type, price_per_night, is_active);
```
**Тип:** составной B-tree.

**Сравнение производительности:**


| Параметр | До оптимизации | После оптимизации |
| :--- | :--- | :--- |
| **План выполнения** | Bitmap Index Scan (`idx_listings_city`) + Filter | Bitmap Index Scan (новый индекс) |
| **Фильтрация** | 1057 строк удалено фильтром | Filter только по `is_active` |
| **Время выполнения** | **12.479 ms** | **2.513 ms** |

**Ускорение:** в 5 раз.

**Вывод:**
Гипотеза подтверждена. Составной индекс позволил СУБД фильтровать данные сразу по нескольким критериям на уровне индекса, не обращаясь к таблице для проверки условий по типу жилья и цене.


### Исследование запроса 2: Сортировка с ограничением

**SQL запрос:**
```sql
EXPLAIN ANALYZE
SELECT u.user_id, u.full_name, COUNT(b.booking_id) AS booking_count
FROM users u
JOIN listings l ON u.user_id = l.host_id
JOIN bookings b ON l.listing_id = b.listing_id
WHERE b.status = 'completed'
GROUP BY u.user_id, u.full_name
ORDER BY booking_count DESC
LIMIT 20;
```

**Гипотеза:**
`Parallel Bitmap Heap Scan` читает избыточный объем данных (21M строк) из-за появления lossy-блоков в оперативной памяти. Создание покрывающего индекса `(status, listing_id)` с включением `booking_id` позволит СУБД использовать `Index Only Scan`, исключив обращение к самой таблице `bookings`.

**Новый индекс:**
```sql
CREATE INDEX idx_bookings_status_listing 
ON bookings (status, listing_id) INCLUDE (booking_id);
```
**Тип:** B-tree + INCLUDE (covering index).

**Сравнение производительности:**


| Параметр | До оптимизации | После оптимизации |
| :--- | :--- | :--- |
| **План выполнения** | Parallel Bitmap Heap Scan + Hash Join | Index Only Scan + Merge Join |
| **Объем чтений** | 21M строк (lossy blocks) | Heap Fetches: 0 |
| **Время выполнения** | **33 230 ms** | **128 ms** |

**Ускорение:** в 260 раз.

**Вывод:**
Гипотеза подтверждена. Переход к `Index Only Scan` радикально сократил количество дисковых операций. Это самый эффективный метод для высоконагруженных таблиц (1.2 млн строк в `bookings`), так как данные извлекаются напрямую из структуры индекса.


### Исследование запроса 3: Альтернативные варианты индексирования

**SQL запрос:**
```sql
EXPLAIN ANALYZE
SELECT b.booking_id, b.start_date, b.end_date, b.total_price, b.status
FROM bookings b
WHERE b.guest_id = 5
AND b.start_date >= '2025-01-01'
AND b.status = 'completed';
```

**Гипотеза:**
Частичный индекс (`Partial Index`) с условием `WHERE status = 'completed'` будет компактнее по размеру и быстрее в поиске, чем классический составной индекс по всем полям.

**Исследуемые варианты:**
*   **Вариант А (составной):** 
    `CREATE INDEX idx_bookings_guest_status_date ON bookings (guest_id, status, start_date) INCLUDE (end_date, total_price);`
*   **Вариант Б (частичный):** 
    `CREATE INDEX idx_bookings_guest_completed ON bookings (guest_id, start_date) INCLUDE (end_date, total_price, status) WHERE status = 'completed';`

**Сравнение производительности:**


| Метрика | До (базовый) | Вариант А (составной) | Вариант Б (частичный) |
| :--- | :--- | :--- | :--- |
| **Тип сканирования** | Index Scan (`idx_guest`) | Index Scan | Index Scan |
| **Фильтрация** | `Filter` по status и дате | `Filter` по status и дате | `Filter` только по дате |
| **Время выполнения** | 2.1 ms | 1.3 ms | **0.2 ms** |

**Ускорение:** в 10 раз (для частичного индекса).

**Вывод:**
Гипотеза подтверждена. Частичный индекс оказался наиболее эффективным: он исключает из своей структуры все записи, не соответствующие условию `completed`, что уменьшает глубину B-дерева и ускоряет поиск. Это оптимальное решение для полей с низкой селективностью или часто используемыми константными фильтрами.


### Исследование запроса 4: Текстовый поиск

**SQL запрос:**
```sql
EXPLAIN ANALYZE
SELECT listing_id, title, city, price_per_night
FROM listings
WHERE title LIKE '%Студия%' OR description LIKE '%центр%';
```

**Гипотеза:**
Стандартный B-tree индекс не поддерживает поиск по маске с ведущим процентом (`LIKE '%...'`). Использование расширения `pg_trgm` и индексов типа `GIN` (триграммы) позволит ускорить поиск произвольных подстрок в текстовых полях.

**Создаваемые индексы:**
```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX idx_listings_title_trgm ON listings USING gin (title gin_trgm_ops);
CREATE INDEX idx_listings_description_trgm ON listings USING gin (description gin_trgm_ops);
```
**Тип:** GIN (триграммы).

**Сравнение производительности:**


| Параметр | До оптимизации | После оптимизации |
| :--- | :--- | :--- |
| **Метод выполнения** | Seq Scan | BitmapOr + Bitmap Index Scan (GIN) |
| **Объем выборки** | 10 000 строк | 10 000 строк |
| **Время выполнения** | **33.7 ms** | **3.0 ms** |

**Ускорение:** в 11 раз.

**Вывод:**
Гипотеза подтверждена. Индексы GIN на основе триграмм эффективно решают задачу полнотекстового поиска подстрок. В отличие от B-tree, они позволяют планировщику отказаться от полного сканирования таблицы (`Seq Scan`) даже при использовании символа `%` в начале поискового запроса.



### Исследование запроса 5: Соединение таблиц (JOIN)

**SQL запрос:**
```sql
EXPLAIN ANALYZE
SELECT b.booking_id, b.start_date, b.end_date, b.total_price,
       g.full_name AS guest_name, l.title AS listing_title,
       u.full_name AS host_name, p.status AS payment_status, r.rating
FROM bookings b
JOIN users g ON b.guest_id = g.user_id
JOIN listings l ON b.listing_id = l.listing_id
JOIN users u ON l.host_id = u.user_id
LEFT JOIN payments p ON b.booking_id = p.booking_id
LEFT JOIN reviews r ON b.booking_id = r.booking_id
WHERE b.status = 'completed' AND b.start_date >= '2025-01-01';
```

**Гипотеза:**
Цепочка `Hash Join` в сочетании с `Bitmap Heap Scan` (обрабатывающим до 19M строк из-за lossy-блоков) является узким местом. Создание индекса по `(status, start_date)` с использованием `INCLUDE` для хранения необходимых колонок ускорит первичную фильтрацию и позволит планировщику сменить стратегию соединения.

**Новый индекс:**
```sql
CREATE INDEX idx_bookings_status_startdate 
ON bookings (status, start_date) 
INCLUDE (end_date, total_price, guest_id, listing_id);
```
**Тип:** B-tree + INCLUDE.

**Сравнение производительности:**


| Параметр | До оптимизации | После оптимизации |
| :--- | :--- | :--- |
| **Метод выполнения** | Hash Join (x5) + Bitmap Heap Scan | Nested Loop + Memoize + Index Scan |
| **Объем чтений** | 19M строк (lossy) | Оптимизированный Index Scan |
| **Время выполнения** | **6 965 ms** | **406 ms** |

**Ускорение:** в 17 раз.

**Вывод:**
Гипотеза подтверждена. Индекс позволил эффективно отсечь данные на раннем этапе. Дополнительный прирост производительности обеспечил узел `Memoize`, который кешировал результаты повторных обращений к таблице `users` (для хостов и гостей), что критично при использовании `Nested Loop` на больших объемах.


### Исследование запроса 6: Негативный сценарий

**SQL запрос:**
```sql
EXPLAIN ANALYZE
SELECT * FROM bookings WHERE total_price > 1000;
```

**Гипотеза:**
Селективность условия близка к 100% (почти все бронирования дороже 1000). В этом случае последовательное сканирование всей таблицы (`Seq Scan`) будет быстрее, чем использование индекса, так как СУБД выгоднее читать блоки данных подряд, чем постоянно переключаться между индексом и таблицей.

**Индекс:**
Не создавался (либо игнорируется планировщиком).

**Результаты тестирования:**


| Параметр | Значение |
| :--- | :--- |
| **Метод выполнения** | Seq Scan |
| **Объем выборки** | 1 200 015 строк |
| **Время выполнения** | **9 363 ms** |

**Вывод:**
Гипотеза подтверждена. При низкой селективности (когда условию удовлетворяет большая часть таблицы) индекс становится бесполезным. Это демонстрирует, что индексирование оправдано только для точечных выборок или условий, значительно ограничивающих результирующий набор данных.

## 6. Влияние индексов на INSERT/UPDATE


| Операция | С индексами | Без индексов | Разница |
| :--- | :--- | :--- | :--- |
| **INSERT 10 000** | 218 ms | 190 ms | +13% медленнее |
| **UPDATE 10 000** | 375 ms | 567 ms | -34% быстрее |

**Вывод:**
Индексы замедляют вставку, так как СУБД требуется дополнительное время на обновление индексных структур. Однако при обновлении (UPDATE) индексы дают значительный прирост производительности, так как радикально ускоряют поиск строк, подлежащих изменению.


## 7. Общие выводы

1. **Правильно подобранный индекс** ускоряет запрос до 260 раз.
2. **Тип индекса должен соответствовать задаче**: B-tree — для сравнений, составной — для нескольких условий, `INCLUDE` — для реализации *Index Only Scan*, частичный — для часто используемых фильтров, GIN — для полнотекстового поиска.
3. **Селективность** — главный критерий эффективности. При селективности <5% оправдано использование индекса, при >15% планировщик обычно выбирает *Seq Scan*.
4. **Планировщик PostgreSQL** самостоятельно выбирает оптимальный план. Задача разработчика — предоставить необходимые индексы и поддерживать актуальную статистику.
5. **Индексы замедляют запись, но ускоряют чтение.** Выбор стратегии индексирования — это всегда компромисс, который зависит от профиля нагрузки (OLTP vs OLAP).
