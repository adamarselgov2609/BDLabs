Лабораторная работа №4

Исследование индексирования и оптимизации запросов в PostgreSQL

Дисциплина: Базы данных
Учебный год: 2025-2026, весна
Вариант: 2 (Airbnb — сервис бронирования жилья)

1. Цель работы

Провести экспериментальное исследование эффективности индексирования в PostgreSQL и обосновать выбор оптимальных индексов для различных типов запросов на основе анализа планов выполнения и характеристик данных.

2. Схема базы данных

База данных сервиса бронирования жилья состоит из 5 таблиц: users, listings, bookings, reviews, payments.

Связи: users 1→N listings (host_id), users 1→N bookings (guest_id), listings 1→N bookings (listing_id), bookings 1→0..1 reviews, bookings 1→0..1 payments.

3. Подготовка тестовых данных

Сгенерированы тестовые данные (generate_data.sql):

users: 10 011 строк
listings: 5 009 строк
bookings: 1 200 015 строк
reviews: 5 (исходные)
payments: 9 (исходных)
Метод: INSERT ... SELECT FROM generate_series(), для bookings — LATERAL для уникальных случайных дат. Триггеры отключались на время генерации. После генерации выполнен ANALYZE.

4. Исследуемые запросы

Запрос 1: Сложный фильтр

Текст запроса:
SELECT * FROM listings
WHERE city = 'Москва'
AND property_type = 'apartment'
AND price_per_night BETWEEN 3000 AND 10000
AND is_active = TRUE;

Гипотеза: существующий индекс idx_listings_city помогает только по городу, остальные условия — Filter. Составной индекс по всем полям WHERE ускорит запрос.

Индекс: CREATE INDEX idx_listings_city_type_price_active ON listings (city, property_type, price_per_night, is_active);
Тип: составной B-tree.

До: Bitmap Index Scan (idx_listings_city) + Filter, 12.479 ms, 1057 строк удалено фильтром.
После: Bitmap Index Scan (новый индекс), Filter только is_active, 2.513 ms.
Ускорение: в 5 раз.
Вывод: гипотеза подтверждена.

Запрос 2: Сортировка с ограничением

Текст запроса:
SELECT u.user_id, u.full_name, COUNT(b.booking_id) AS booking_count
FROM users u
JOIN listings l ON u.user_id = l.host_id
JOIN bookings b ON l.listing_id = b.listing_id
WHERE b.status = 'completed'
GROUP BY u.user_id, u.full_name
ORDER BY booking_count DESC
LIMIT 20;

Гипотеза: Parallel Bitmap Heap Scan читает 21M строк из-за lossy-блоков. Индекс (status, listing_id) INCLUDE (booking_id) даст Index Only Scan.

Индекс: CREATE INDEX idx_bookings_status_listing ON bookings (status, listing_id) INCLUDE (booking_id);
Тип: B-tree + INCLUDE (covering index).

До: Parallel Bitmap Heap Scan, 21M строк, lossy блоки, Hash Join, 33 230 ms.
После: Index Only Scan, Heap Fetches: 0, Merge Join, 128 ms.
Ускорение: в 260 раз.
Вывод: гипотеза подтверждена. Index Only Scan — самый эффективный метод.

Запрос 3: Альтернативные варианты индексирования

Текст запроса:
SELECT b.booking_id, b.start_date, b.end_date, b.total_price, b.status
FROM bookings b
WHERE b.guest_id = 5
AND b.start_date >= '2025-01-01'
AND b.status = 'completed';

Гипотеза: частичный индекс WHERE status='completed' будет компактнее и быстрее составного.

Вариант А (составной): CREATE INDEX idx_bookings_guest_status_date ON bookings (guest_id, status, start_date) INCLUDE (end_date, total_price);
Вариант Б (частичный): CREATE INDEX idx_bookings_guest_completed ON bookings (guest_id, start_date) INCLUDE (end_date, total_price, status) WHERE status = 'completed';

До: Index Scan (idx_bookings_guest), Filter по status и дате, 2.1 ms.
Вариант А: Index Scan, Filter по status и дате, 1.3 ms.
Вариант Б: Index Scan, Filter только по дате, 0.2 ms.
Ускорение: в 10 раз (частичный).
Вывод: частичный индекс эффективнее. Меньше размер → быстрее поиск.

Запрос 4: Текстовый поиск

Текст запроса:
SELECT listing_id, title, city, price_per_night
FROM listings
WHERE title LIKE '%Студия%' OR description LIKE '%центр%';

Гипотеза: B-tree не работает с LIKE '%...%'. GIN + pg_trgm ускорит поиск подстроки.

Индексы:
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_listings_title_trgm ON listings USING gin (title gin_trgm_ops);
CREATE INDEX idx_listings_description_trgm ON listings USING gin (description gin_trgm_ops);
Тип: GIN (триграммы).

До: Seq Scan (10 000 строк), 33.7 ms.
После: BitmapOr + Bitmap Index Scan (GIN), 3.0 ms.
Ускорение: в 11 раз.
Вывод: гипотеза подтверждена. GIN эффективен для поиска подстрок.

Запрос 5: Соединение таблиц (JOIN)

Текст запроса:
SELECT b.booking_id, b.start_date, b.end_date, b.total_price,
g.full_name AS guest_name, l.title AS listing_title,
u.full_name AS host_name, p.status AS payment_status, r.rating
FROM bookings b
JOIN users g ON b.guest_id = g.user_id
JOIN listings l ON b.listing_id = l.listing_id
JOIN users u ON l.host_id = u.user_id
LEFT JOIN payments p ON b.booking_id = p.booking_id
LEFT JOIN reviews r ON b.booking_id = r.booking_id
WHERE b.status = 'completed' AND b.start_date >= '2025-01-01';

Гипотеза: цепочка Hash Join + Bitmap Heap Scan (19M строк) — узкое место. Индекс по (status, start_date) с INCLUDE ускорит первичный фильтр.

Индекс: CREATE INDEX idx_bookings_status_startdate ON bookings (status, start_date) INCLUDE (end_date, total_price, guest_id, listing_id);
Тип: B-tree + INCLUDE.

До: Hash Join ×5, Bitmap Heap Scan (19M lossy), 6 965 ms.
После: Nested Loop + Memoize, Index Scan, 406 ms.
Ускорение: в 17 раз.
Вывод: гипотеза подтверждена. Memoize дополнительно ускорил повторные обращения к users.

Запрос 6: Негативный сценарий

Текст запроса:
SELECT * FROM bookings WHERE total_price > 1000;

Гипотеза: селективность ~100% → Seq Scan быстрее индекса. Индекс не нужен.

Индекс не создавался.

Результат: Seq Scan, 1 200 015 строк, 9 363 ms.
Вывод: гипотеза подтверждена. При низкой селективности индекс бесполезен.

7. Общие выводы

Правильно подобранный индекс ускоряет запрос до 260 раз.
Тип индекса должен соответствовать задаче: B-tree для сравнений, составной для нескольких условий, INCLUDE для Index Only Scan, частичный для частых фильтров, GIN для текста.
Селективность — главный критерий. <5% — индекс, >15% — Seq Scan.
Планировщик PostgreSQL сам выбирает план. Задача разработчика — дать ему индексы и актуальную статистику.
Индексы замедляют запись, но ускоряют чтение. Компромисс зависит от профиля нагрузки.
